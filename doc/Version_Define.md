
核心定义：语义化版本控制 (SemVer)
这是目前最主流、最受推崇的版本号定义规范，被广泛应用于各种编程语言和包管理器中。它的核心思想是通过版本号本身来传达关于软件变更的准确信息。

版本号格式为：主版本号.次版本号.修订号（例如：1.2.3），每个部分的递增都有明确的规则：

主版本号 (Major)：当你做了不兼容的 API 修改时递增。这意味着这个版本包含重大变更，旧版本的使用者可能需要进行代码修改才能升级。例如，从 1.5.3 变为 2.0.0。

次版本号 (Minor)：当你以向下兼容的方式添加了新功能时递增。新增功能不会破坏现有代码，用户可以放心更新。例如，从 1.5.3 变为 1.6.0。

修订号 (Patch)：当你做了向下兼容的问题修正（如Bug修复）时递增。这些通常是一些小补丁，不影响软件的功能和接口。例如，从 1.5.3 变为 1.5.4。

在初始开发阶段（版本号以 0 开头，如 0.y.z），软件API不稳定，可以随时发生变更。通常认为，当软件被部署到生产环境或拥有稳定的API时，就应该发布 1.0.0 版本。

开发阶段标识：测试版与正式版
在版本号中，我们常会看到 -alpha、-beta、-rc 这样的后缀，它们用于标识软件在正式发布前所处的不同测试阶段。它们的稳定性和成熟度通常是递增的：

Alpha版：内部测试版。这是最早期的版本，主要目标是实现软件功能。通常Bug较多，仅供开发团队内部或少数专业测试人员使用。

Beta版：公开测试版。相对于Alpha版，已有了很大改进，消除了严重的错误。但软件可能仍存在一些缺陷，需要经过更广泛的用户测试来发现。

RC版 (Release Candidate)：发布候选版。这个版本已经相当成熟，基本不存在会导致错误的重大Bug，与即将发行的正式版相差无几。这是正式版发布前的最后一个测试阶段。

Release / Stable版：正式版/稳定版。这是最终交付给用户使用的版本，经过了充分的测试，具有最高的稳定性和可靠性。
